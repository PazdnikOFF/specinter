# Новый портал specinter — архитектура и предложение

> Цель: **автономный продающий портал запчастей китайской спецтехники в России** с
> ИИ‑оператором, автоматическим приёмом прайсов, подбором по артикулу, расчётом сроков
> доставки, приёмом платежей физлиц, прямой интеграцией с 1С:УНФ и ЭДО Диадок.
> UI — в лучших традициях Apple. Всё в Docker. Разработка в `new/`.

## 1. Принципы

1. **Search‑first**: главный сценарий — ввёл артикул/название/аналог → мгновенно нашёл → купил.
2. **Автономность**: бот и портал закрывают весь путь; оператор‑человек — только в крайних случаях.
3. **Артикул производителя — первичный ключ подбора** (не внутренний код 1С, как в legacy).
4. **1С — источник истины по учёту**; интеграция **только прямая (API), без файлов**.
5. Чистая доменная модель в Postgres; legacy‑данные мигрируются, legacy‑код — нет.

## 2. Технологический стек

| Слой | Технология | Почему |
|---|---|---|
| Frontend | **Next.js (App Router) + TypeScript + Tailwind** | SSR/ISR для SEO каталога, PWA, Apple‑грейд UI |
| Instant search | **Meilisearch** (или Typesense) | поиск‑как‑печатаешь по артикулу/аналогам, опечатки |
| Backend API | **Python + FastAPI** | совпадает со стеком пользователя (AI OS, PACS), быстрый async |
| БД | **PostgreSQL** + `pg_trgm` | нечёткий поиск по артикулу, надёжность, FTS |
| Кэш/очереди | **Redis** + **Celery** (или ARQ) | фоновые задачи: парсинг почты, синк 1С, пересчёт цен |
| Хранилище | **MinIO (S3)** | изображения товаров и схем каталогов |
| ИИ‑оператор | **Claude (Anthropic)** — агент с инструментами | подбор, поиск, оформление; эскалация к человеку |
| Инфраструктура | **Docker Compose** (dev) → возможен Swarm/k8s (prod), Nginx | автономный self‑hosted деплой |

## 3. Доменная модель (Postgres, ядро)

- `categories` — дерево (materialized path); из `it_tree`.
- `products` — карточка‑артикул: `manufacturer_article` (нормализованный), `name`, `brand`, `machine_models[]`, описание, SEO; из `it_b_catitem`.
- `product_images` — из `it_b_goodimage` / `img/`.
- `analogs` — кросс‑ссылки артикулов (`product_id`, `analog_article`, `source`); из `it_b_aarts` (ценный актив).
- `suppliers` — поставщик: название, e‑mail отправителя прайса, **город** (для расчёта доставки), правила наценки, профиль парсинга.
- `supplier_price_profiles` — маппинг колонок XLSX, смещение шапки, наценка, валюта — **на каждого поставщика**.
- `supplier_prices` — строки прайса: `supplier_id`, `article`, `price`, `qty`, `city`, `received_at`; **с историей**.
- `offers` — витринное предложение по товару: лучший/выбранный поставщик, итоговая цена, наличие, срок.
- `warehouses` — наш склад(ы) (Екатеринбург).
- `customers` — физ/юрлица + реквизиты + код 1С; из `it_b_user3`. `garage` (сохранённая техника) — переносим.
- `orders` / `order_items` — заказы, статусы, привязка к оплате/документам/1С.
- `payments` — транзакции эквайринга.
- `documents` — счёт/УПД/накладная: id в 1С, ссылка на **печатную форму (ПФ)**, статус ЭДО.

Матчинг артикула: нормализация (регистр, пробелы, разделители `-/.`), точное совпадение →
таблица `analogs` → нечёткий `pg_trgm` с порогом уверенности → **очередь ручной проверки**.

## 4. Конвейер приёма прайсов (замена `xml.php`)

```
IMAP‑почтовый ящик (poll по расписанию, Celery)
   → определение поставщика по адресу отправителя
   → выбор supplier_price_profile (маппинг колонок, шапка, наценка, город)
   → парсинг XLSX (openpyxl) → нормализация артикула
   → матчинг к products по manufacturer_article (+ analogs + fuzzy)
   → upsert в supplier_prices С ИСТОРИЕЙ (не TRUNCATE!)
   → пересчёт offers (лучшее предложение с учётом цены/наличия/срока)
   → неуверенные матчи → очередь модерации в админке
```

Что исправляем против legacy: поартикульный матчинг; per‑supplier профили и наценки;
приём с почты; история цен; отсутствие взаимного затирания поставщиков; безопасные запросы.
Ручная загрузка XLSX в админке остаётся как резервный путь.

## 5. Расчёт сроков доставки

- **Поставщик → наш склад** (срок пополнения): `supplier.city` → склад (Екатеринбург).
  Матрица транзита по API перевозчиков (ПЭК / Деловые линии / СДЭК) или статическая таблица.
- **Склад → клиент** (срок отгрузки): геолокация клиента (если подтвердил) → геокодинг
  (DaData/Yandex) → API перевозчика.
- В карточке: «в наличии → N дней» против «под заказ → M дней» (пополнение + отгрузка).

## 6. Платежи физлиц

- **ЮKassa** или **Т‑Банк (Тинькофф)**: карты + **СБП**, вебхуки статусов, чеки 54‑ФЗ.
- Юрлицам — **счёт создаётся напрямую в 1С:УНФ** (см. ниже), не PDF‑заглушкой.

## 7. Прямая интеграция с 1С:УНФ (без файлов)

Требование: **только прямое API‑взаимодействие**, автоматическое создание всех документов и
получение печатных форм (ПФ), результатов, данных из 1С напрямую.

- **Транспорт:** стандартный OData‑интерфейс 1С (`/odata/standard.odata`) и/или **HTTP‑сервисы**
  1С (расширение конфигурации УНФ) для операций, которых нет в OData (проведение, печать ПФ).
- **Создаём в 1С напрямую:** Заказ покупателя, Счёт на оплату, Расходная накладная — по факту заказа.
- **Получаем из 1С напрямую:** номенклатура/цены/остатки, статусы оплаты и отгрузки,
  **печатные формы (PDF)** счёта/УПД, номера и проведённость документов.
- **Клиенты/контрагенты:** синхронизация по коду 1С (перенос из `it_b_user3`).
- Реализуется как отдельный сервис `unf-connector` (FastAPI‑клиент к 1С), очередь на повтор при недоступности 1С.

> Если у 1С:УНФ ограничен OData (например, облако 1cfresh) — операции проведения и печати ПФ
> выносятся в HTTP‑сервисы расширения конфигурации. Это уточняем по вашей инсталляции УНФ
> (коробка/сервер/облако, версия, доступ к публикации на веб‑сервере).

## 8. ЭДО Диадок

- API **Контур.Диадок**: автоматическое получение закрывающих (УПД/счёт‑фактура/акт),
  привязка к заказу/клиенту, отслеживание статуса подписания.
- Триггер — событие отгрузки/проведения в 1С (через `unf-connector`).

## 9. ИИ‑оператор (Telegram / WhatsApp / MAX)

- **Единое ядро‑агент** на **LangGraph** (`assistant ↔ tools`), LLM **провайдер‑агностично** через
  OpenAI‑совместимый API — **Grok** (`api.x.ai/v1`) или **Hermes** (OpenRouter/vLLM), меняется env.
- Адаптеры каналов: **Telegram, MAX (РФ), Instagram Direct, WhatsApp** — одна логика, четыре транспорта.
- **Автопостинг в Instagram**: бот сам публикует рилсы/фото‑промо из каталога (Content Publishing API);
  платная реклама — Marketing API (нужен рекламный кабинет и бюджет).
- Скиллы вычищают конфиденциальное (поставщики/закупка), строгие guardrails — только продажа запчастей.
- **WhatsApp (решение заказчика):** номер сейчас привязан к обычному приложению на телефоне. Варианты:
  - *Официально* — WhatsApp Cloud API (WABA): номер переносится в Business Platform и **перестаёт
    работать в обычном приложении**. Надёжно, без риска блокировки, есть шаблоны сообщений.
  - *Мост WhatsApp Web (Baileys)* — подключение по QR как «WhatsApp Web», телефон остаётся;
    **неофициально, риск бана номера**. Приемлемо для MVP/пилота.
  Реализуем адаптер так, чтобы транспорт WhatsApp можно было заменить без изменения ядра‑агента.
- **Инструменты агента:** поиск по каталогу/артикулу, подбор аналогов, цена+наличие,
  расчёт срока доставки, оформление заказа, выставление счёта/оплаты, передача оператору.
- **RAG** по каталогу и справочным материалам; ответы с ссылками на карточки.
- **Автономность:** человек подключается только при низкой уверенности агента или явном запросе.
- Согласуется с вашим проектом **AI Engineering OS** (оркестрация моделей/Skills).

## 13. Мои дополнительные предложения

- **Подбор по технике**: «выбери модель (Shantui SD16) → узел → деталь» + поиск по VIN/серийнику.
- **«Гараж»** клиента (уже есть в legacy) — сохранённая техника и история заказов, подсказки бота.
- **RFQ / запрос цены** на отсутствующие позиции → авто‑рассылка поставщикам.
- **Уведомления** о снижении цены / появлении в наличии (бот + e‑mail).
- **Рейтинг надёжности поставщика** (fill‑rate, отклонение сроков) для выбора offer.
