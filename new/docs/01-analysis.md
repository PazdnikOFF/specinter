# Анализ текущего сайта specinter.ru

> Источник: `legacy/` (выгрузка с хостинга) + `legacy/specinter.dump.sql` (БД `c26864_specinter_ru`, MariaDB 10.5).
> Компания: **ООО «СПЕЦИНТЕР»**, Екатеринбург, пер. Шоферов 11. Запчасти для китайской спецтехники.

## 1. Что продаётся и позиционирование

Запчасти для спецтехники: бульдозеры (Shantui SD16/22/32, Zoomlion, Komatsu), погрузчики
(XCMG, SDLG, Lonking, LiuGong), экскаваторы (Doosan, Hyundai, Hitachi), грузовики
(Howo, Shacman, Dong Feng, FAW), двигатели (Weichai WP10/WP12, SC9D…), фильтры, форсунки, трансмиссии.

УТП сайта: «в наличии всегда», собственный склад, прямые поставки без перекупов,
доставка ПЭК / Деловые линии / СДЭК, оплата счётом или картой с НДС.
Навигация: ЗАПЧАСТИ · КАТАЛОГИ · ОПЛАТА И ДОСТАВКА · О НАС · УСЛУГИ · КОНТАКТЫ.

## 2. Технологический стек (legacy)

- **Самописная PHP‑CMS** (префикс таблиц `it_`, шаблонизатор ETS, кэш phpFastCache).
- Смешаны `mysql_*` и `mysqli_*`, **повсеместные SQL‑инъекции** (строковая конкатенация).
- Остатки Bitrix (эквайринг УБРиР по пути `/bitrix/modules/...`).
- Фронт: jQuery + зоопарк плагинов (slick, arcticmodal, elevateZoom, kladr, formstyler).
- Хостинг‑привязка: жёсткие пути `/home/c26864/specinter.ru/www/`.

**Вывод:** движок неремонтопригоден для новых задач. Ценность legacy — **данные и бизнес‑логика**, а не код.

## 3. Модель каталога (как на exist.ru)

| Таблица | Роль | Объём (AUTO_INCREMENT) |
|---|---|---|
| `it_tree` | Дерево категорий (nested set: `left_key`/`right_key`), корень каталога id=40 | ~1676 |
| `it_b_catitem` | **Карточка‑артикул** (товар как «категория»): `art`, `name_rus/eng`, html, изображения, SEO. FULLTEXT по `art,name_rus,name_eng,html` | ~20500 |
| `it_b_ablock` | Связка карточка↔категория (FK `good_id`→catitem, CASCADE) | ~38600 |
| `it_b_variant` | **Торговое предложение** (то, что реально покупается): `price`, `quantity`, `art`, `manufacture`, `xlsx_id`(=код 1С), `time` | ~2668 |
| `it_b_aarts` | **Аналоги / дубли артикулов**: `art`, `good_id_arts` (привязка к карточке) | ~5988 |
| `it_b_user3` | Клиенты + реквизиты юрлиц (`inn,kpp,bank,bik,rschet,korschet`), `garage`, код 1С | ~1388 |
| `it_b_order` / `it_b_orderitem` | Заказы (используются слабо) | 1 / ~1619 |
| `it_suppliers` | Стейджинг разобранного прайса (сериализованные строки), **TRUNCATE при каждом импорте** | — |

Логика: **товар = артикул‑карточка** с набором торговых предложений (variant) и списком
аналогов (aarts). Это ценная и правильная для запчастей модель — переносим в новый портал.

## 4. Парсер прайсов (`legacy/backend/inc/modules/xml.php`)

Как работает сейчас:

1. Админ **вручную** загружает XLSX через форму (`$_FILES['file']`).
2. Читается лист 0, пропускаются первые 7 строк, **жёстко зашитые колонки**:
   `0=maker, 1=name, 2=art, 4=price, 5=quantity, 6=code(код 1С)`.
3. `TRUNCATE it_suppliers` → строки пишутся сериализованными (`serialize()`).
4. Сопоставление (`read()`, пачками по 500 через HTTP‑редирект):
   - матчинг **по коду 1С**: `it_b_variant WHERE xlsx_id = code`;
   - наценка: `ceil(price * 1.15 / 0.5) * 0.5` (15 % + округление до 0.5);
   - нашёл → UPDATE variant + catitem; не нашёл → создаёт новые catitem+ablock+variant (parent 245/451).

**Проблемы, которые надо устранить в новом портале:**

- ⚠️ Матчинг идёт **по внутреннему коду 1С**, а не по **артикулу производителя** — а задача требует сопоставления именно по артикулу.
- Один жёсткий формат колонок и одна наценка для всех; поле `supplier` есть, но per‑supplier конфига нет.
- `TRUNCATE` стирает прайсы **всех** поставщиков при загрузке одного.
- Нет приёма прайсов **с почты** (только ручная загрузка).
- Нет истории цен, дедупликации, города поставщика → **невозможен расчёт доставки**.
- SQL‑инъекции, редирект‑в‑цикле, единый лист/индекс — хрупко.

Формат строки прайса (из `it_suppliers.data`): `maker, name, art, price, quantity, code`.

## 5. Обмен с 1С (`legacy/backend/inc/modules/import1c.php`)

Двусторонний, формат близкий к CommerceML:

- `getXml` — **экспорт клиентов** в 1С: `Code, Address, Email, Telephone`.
- `importClass` — **импорт каталога** из `images/from1c.xml` (дерево `CreationParents` + позиции).

То есть сегодня **1С — мастер каталога и кодов клиентов**, но обмен **файловый** (XML на диске).
В новом портале файловый обмен исключается — только прямая API‑интеграция с 1С:УНФ (см. `02-architecture.md` §7).

## 6. Оплата и документы

- Эквайринг **УБРиР** через curl + клиентский сертификат (`payment.php`), завязан на путь Bitrix — вероятно, нерабочий/устаревший.
- Счета юрлицам — генерация PDF (dompdf) и XLSX (PHPExcel), реквизиты зашиты в коде (ВТБ, счёт 4070…350).
- **ЭДО отсутствует.**

## 7. Чего сейчас нет (и требуется по ТЗ)

- Автоприём прайсов с почты и per‑supplier парсинг.
- Матчинг по артикулу производителя (+ аналоги, + нечёткий поиск).
- Расчёт срока поставки: город поставщика → наш склад, и склад → геолокация клиента.
- Приём платежей физлиц (карты/СБП).
- Обмен с 1С:УНФ для счетов юрлицам.
- ЭДО Диадок для закрывающих документов.
- Бот‑оператор в Telegram / WhatsApp / MAX.
- Автономность + админка с метриками.

## 8. Что переносим из legacy (ценные активы)

- **Изображения** товаров (`img/`, `it_b_goodimage`) и схемы каталогов.
- **Каталог**: карточки‑артикулы, категории, торговые предложения.
- **Аналоги** (`it_b_aarts`) — редкий и ценный актив для подбора запчастей.
- **База клиентов** с реквизитами юрлиц и кодами 1С.
- Контракт обмена с 1С (коды номенклатуры/клиентов) как отправная точка для УНФ.
