# Методология хранения и загрузки каталога и прайсов

Документ фиксирует, КАК данные раскладываются по таблицам новой структуры и как
их загружать. Дополняет `02-architecture.md` (раздел «Данные»). Появился после
загрузки ATQ-каталога `all.xls` и прайса «МСК ОПТ» (16.07.2026).

## 1. Разделение сущностей (что где хранится)

| Таблица | Смысл | Ключевое правило |
|---|---|---|
| `products` | Карточка = **1 реальный OEM-артикул производителя** | `manufacturer_article` — ТОЛЬКО чистый номер, без брендовых суффиксов и SKU поставщика. Дедуп по `normalized_article`. |
| `analogs` | Кросс-номера, аналоги и варианты установки | Чужие OEM, внутренние SKU-кроссы прайса; `linked_product_id` — если у кросса есть своя карточка. `source` помечает происхождение, `relation_type` — смысл связи (см. ниже). |
| `suppliers` | Поставщики | Наценка `markup_percent`, округление `price_round`, срок `delivery_days`. |
| `supplier_prices` | **Сырьё** — каждая строка прайса, с историей | `matched_product_id` + `match_method`; хранит `raw` (все колонки прайса). |
| `offers` | **Торговое предложение** по товару | Пересчёт из `supplier_prices`; цена с наценкой, наличие, срок. |

Товары — стабильны; прайсы/цены — часто меняются. Поэтому строго раздельно.

### 1.1. `analogs.relation_type` — два разных смысла связи

На старом сайте связи товар↔товар жили в ДВУХ таблицах и выводились разными блоками
карточки. При переносе они схлопнуты в одну таблицу с разделителем по типу:

| `relation_type` | Legacy-источник | Блок карточки | Смысл |
|---|---|---|---|
| `analog` (дефолт) | `it_b_aarts` | «Аналоги» | Взаимозаменяемо — можно ставить вместо |
| `install_option` | `it_b_aarts2` | «Возможные варианты установки» | Встаёт как вариант, но **не** прямая замена |

**Правило: `install_option` НЕ участвует в подборе.** Он исключён из поиска
(`search.reindex`), из `/quick-resolve` и из `/products/by-article` — иначе запрос по
номеру начнёт выдавать деталь, которая запрошенному артикулу не замена. Он виден
только отдельным блоком в карточке. Там же — и «от N ₽» по группе: `group_min_price`
считается по `analog`, но НЕ по `install_option`.

Не путать с `position_variants` — те вычисляются на лету из совпадения `position`
в `product_categories` и в БД не хранятся.

## 2. Правила загрузки

### Каталог (наименование + артикул + кроссы, БЕЗ цен)
1. `manufacturer_article` = **чистый OEM-артикул**. Брендовые суффиксы (`-ATQ`, `-AT`,
   `-Q` у ATQ/AVTO-TECH) **срезать**. Дефисы ВНУТРИ номера (`5301625-C0100`) сохранять.
2. **НЕ** использовать первый «кросс» как источник артикула — в этом файле кросс часто
   является номером ДРУГОГО производителя (проверено: совпадал лишь у ~1000 из 14244).
3. Кроссы → `analogs` (по одному токену на строку), дедуп по `(product_id, normalized_article)`,
   без self-ссылки, `linked_product_id` при совпадении с карточкой.

### Прайс (с ценой и остатком)
1. Для матча `supplier_prices.article` = **чистый OEM-артикул** (та же нормализация,
   что у товаров) → `exact_article`. НЕ брать SKU поставщика (напр. `AT0003`) — он не матчится.
2. Приоритет матча: `exact_article` → `analog` → `cross` → `fuzzy` (последний — только в модерацию).
3. `offers` — пересчёт с наценкой поставщика: `ceil(price*(1+markup/100)/price_round)*price_round`.

## 3. Механизм тиров/брендов (ЦЕЛЕВОЙ — пока НЕ реализован)

**Проблема.** Один реальный OEM-артикул часто предлагается под РАЗНЫМИ брендами с разной
ценой/качеством. Пример `DZ14251440060`:

| Производитель (бренд) | Артикул в прайсе | Цена |
|---|---|---|
| AVTO-TECH (premium, `-ATQ`) | `DZ14251440060-ATQ` | 4800 |
| CHINA (noname) | `DZ14251440060` | 3564 |
| SHACMAN (OEM) | `DZ14251440060` | 3177 |

В прайсе **946** реальных артикулов имеют >1 бренда. OEM SHACMAN здесь ДЕШЕВЛЕ премиума —
значит «показать минимальную цену» скрывает от клиента осмысленный выбор.

**Неправильные способы (отвергнуты):**
- Кодировать тир в артикуле (`...-ATQ`) — ломает матчинг и поиск по OEM-номеру.
- Схлопывать в один offer по минимуму — теряет бренды и выбор (это текущее поведение, см. §4).

**Правильный механизм — БРЕНД на уровне offer** (колонка «Производитель» прайса — это и есть ось тира):
- Добавить `offers.brand text` (из col «Производитель»); при желании уточнять линию суффиксом
  (`-ATQ` → «AVTO-TECH QUALITY»).
- Уникальность: `UNIQUE (product_id, supplier_id, brand) WHERE source='price'`
  вместо текущего `(product_id, supplier_id)`.
- `ingest`: `DISTINCT ON (matched_product_id, brand)` — по одному offer на бренд (внутри бренда — дешевле).
- Витрина: блок «Предложения» и карточка показывают бренды отдельными строками
  (бренд + цена + наличие + срок); «от N ₽» = **min по брендам**.
- `products.brand` больше НЕ заполнять по самому частому maker (бренд ушёл на offer);
  оставить для реального бренда производителя детали, если он однозначен.

Порядок внедрения (отдельная задача): миграция схемы `offers` → доработка `prices.ingest`
(маппинг brand, distinct on brand) → UI карточки/страницы товара → reindex.

## 4. Текущее состояние (16.07.2026, ЛОКАЛЬНАЯ dev-БД)

- Каталог и прайс «МСК ОПТ» загружены по §2. Тиры **схлопнуты в один offer по минимуму**
  (текущая логика `ingest` с `UNIQUE (product_id, supplier_id)`), бренды пока не различаются.
- Механизм §3 — **не реализован**; текущую логику `ingest` менять НЕ требуется до отдельной задачи.
- Данные живут только в локальной БД (том `pgdata`); на прод не переносились.
